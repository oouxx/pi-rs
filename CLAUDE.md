# CLAUDE.md — pi (TypeScript → Rust) 复刻项目

本文件是本项目的固定工作规范，Claude Code 在本仓库内工作时必须遵守。

## CLAUDE.md 是建议性的，不是强制的

Claude Code 官方文档明确说明：CLAUDE.md 里的规则是 advisory（建议性
上下文），Claude 会读、通常会照做，但不保证 100% 遵守——尤其是文件
写得太长时，重要规则会被淹没，Claude 会直接忽略一部分。真正"零例外"
的规则，要靠 **hooks**（在特定事件触发时自动跑脚本，确定性执行，不
依赖模型当时的状态）。

本项目已经在 `.claude/hooks/` 下配了三个 hook，对应这份文档里最不能
妥协的几条：

| Hook                            | 触发时机                                            | 对应 CLAUDE.md 规则                                                                                                |
| ------------------------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `check-unwrap.sh`               | 每次 Edit/Write 修改 `.rs` 文件后                   | 非测试代码禁止 `.unwrap()`/`.expect()`                                                                             |
| `block-git-danger.sh`           | 每次执行 Bash 前                                    | 并行 workflow 中禁止 `git stash`/`reset --hard` 等破坏性命令                                                       |
| `verify-before-stop.sh`         | 每轮对话结束前（仅当本轮改动了 `.rs` 文件时才触发） | `cargo clippy` 零警告 + `cargo test --workspace` 通过才算完成                                                      |
| `check-contract-table.sh`       | 每次执行 `git commit` 前                            | 公开接口（`pub fn`/`struct`/`enum`/`trait`）有改动时，`CONTRACT_ALIGNMENT.md` 必须同步提交，不能只改代码不改对照表 |
| `check-contract-consistency.sh` | 每轮对话结束前                                      | `CONTRACT_ALIGNMENT.md` 里"是否一致=否"的行必须引用 `DEVIATIONS.md`，不能留一个没处理完的"否"                      |

**结论：** 这份 CLAUDE.md 里凡是标了"必须""禁止""不可跳过"的规则，
默认按"建议"对待就已经不够了；如果发现 Claude 反复违反同一条，第一
反应不是把这句话写得更用力（加粗、加"IMPORTANT"），而是问"这条能不
能写成 hook"。文档剩下的部分主要覆盖 hook 覆盖不到的、需要判断力的
规则（比如三阶段顺序、`PORTING.md` 映射是否合理、对抗式复核的深度），
这些交给 hook 太死板，还是要靠 Claude 理解上下文来执行。

## 项目背景

将 `earendil-works/pi` monorepo（TypeScript）移植为 Rust workspace。
原则：不追求逐行翻译，追求**行为一致**——尤其是状态机语义和公开 API 的
输入输出契约。

ts源码在当前项目根目录的同级pi目录下

## 工作流：三阶段，禁止跳步

### 阶段一：架构分析（只产出文档，不写代码）

**产出物落盘为 `PORTING.md`（仓库根目录，跨 crate 共用一份），而不是
只在对话里说一遍就过去。** 原因：后续每一次"这段该怎么翻译"的判断，
都应该查 `PORTING.md` 里已经定好的映射规则，而不是每个模块重新拍板，
否则同一种 TS 模式在不同文件里会被翻成不一致的 Rust 写法。

`PORTING.md` 至少包含一张"模式映射表"：

| TS 模式                      | Rust 对应写法                                                       | 备注                    |
| ---------------------------- | ------------------------------------------------------------------- | ----------------------- |
| `field?: T`                  | `Option<T>`                                                         |                         |
| `type A = X \| Y`            | `enum A { X(..), Y(..) }`                                           |                         |
| `interface Provider { ... }` | `trait Provider { ... }`                                            | trait-first，见代码规范 |
| `?? defaultValue`            | 显式 `match` / `unwrap_or_else`（非 `unwrap_or`，见下方"高危陷阱"） | 禁止用会掩盖异常的写法  |

对**每一个涉及的源码目录**，分别输出以下四类内容。禁止使用"大致"
"类似"等模糊词；每一项都要能在原仓库源码中定位到具体文件/行号。

**1. 模块列表和职责**（逐文件，不是逐目录）

| 文件路径 | 导出的主要符号 | 职责（一句话） | 是否为公开 API（被其他 crate 引用） |
| -------- | -------------- | -------------- | ----------------------------------- |

**2. 核心数据结构**（对每个核心 type/interface）

- 原始 TS 定义（贴代码块）
- 字段逐个说明：类型、是否 optional、默认值、不变量（invariant）
- Rust 映射方向：
  - `interface` → `struct` 还是 `trait`？
  - `field?: T` → `Option<T>`
  - `type A = X | Y` → `enum`，列出每个 variant
  - 涉及 `any`/动态类型的地方单独标注为高风险点

**3. 模块间依赖关系**

- 文字描述 import 关系（谁 import 谁），覆盖所有跨文件依赖
- 是否存在循环依赖；若有，说明 TS 侧如何解耦（如用 interface），
  Rust 侧对应用 trait 还是拆子 crate 解决

**4. 对外接口**

- 每个公开函数/方法的完整签名（参数名、类型、返回类型、是否 async、
  是否可能 throw/reject 及触发条件）
- 事件流/回调接口（如 `agent.subscribe(...)`）单独列出所有事件
  variant 和触发时机

**验收标准：** 产出物必须让人不看原仓库代码，就能推导出 Rust 侧的
类型定义骨架。做不到就说明分析深度不够，需要补充。

#### 试跑（正式铺开前，先挑 2-3 个文件验证 `PORTING.md`）

不要拿到 `PORTING.md` 就直接对整个目录批量翻译。先选 2-3 个有代表性
的文件（挑一个类型定义密集的、一个逻辑复杂的），走完整套阶段二+三
流程。如果这几个文件暴露出 `PORTING.md` 的映射规则有问题，先回去改
`PORTING.md`，再铺开到其余文件——避免用错误的映射规则批量翻译一遍，
之后又要全部推倒重来。

#### 已知高危陷阱：TS 和 Rust "长得像但行为不同"的模式

这类问题最危险，因为代码能编译、也不会立刻报错，只在特定输入下才
表现出差异。分析阶段发现以下模式必须在 `PORTING.md` 里单独标注，
不能默认"看着差不多就直接翻":

| 模式            | TS 行为                                                                           | Rust 天真翻译的坑                                                                              | 正确做法                                                             |
| --------------- | --------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 短路默认值      | `a ?? (sideEffect())`，`a` 非 null 时 `sideEffect()` 不执行                       | `a.unwrap_or(side_effect())` 的参数是**立即求值**的，副作用永远执行                            | 用 `unwrap_or_else(\|\| side_effect())`                              |
| 数值截断        | `Math.trunc()` 向零取整，负数、小数场景要单独确认原意图是"向零"还是"向下"         | Rust 的 `as i64` 转换也是向零截断，但如果原意图其实是 `floor`，两边会在负数上产生不同结果      | 逐个数值转换点确认原意图，不要假设 `as` 和 TS 的转换语义天然一致     |
| 数组/字符串越界 | JS 访问越界下标返回 `undefined`，不 panic                                         | 直接 `slice[i]` 在 Rust 里越界会 panic；原 TS 代码里可能存在"依赖越界返回 undefined"的隐式逻辑 | 显式用 `.get(i)` 返回 `Option`，逐处确认原代码是否依赖了这种隐式行为 |
| async 并发顺序  | `Promise.all([...])` 的调度细节和 `tokio::join!` / `futures::join_all` 不完全一致 | 状态机对时序敏感的地方（如事件流顺序）直接翻译可能改变事件到达顺序                             | 涉及并发顺序的模块必须做阶段三里的"事件序列对齐"，不能只看单测通过   |

如果在具体文件里发现新的此类陷阱，随时补进 `PORTING.md` 的这张表，
不要只在当次对话里提一句就过去。

### 阶段二：按模块重写（Rust 实现）

**crate 顺序：** 先 `pi-ai`（无内部依赖）→ 再 `pi-agent-core`
（依赖 `pi-ai`）→ 依此类推。

**同一 crate 内部：** 先写被依赖最多的底层模块（如 message 类型
定义），再写上层的 agent loop / provider 适配。

**每个模块内部的写作顺序（强制，不可跳过或颠倒）：**

1. **类型定义**：struct/enum/trait，先不写方法体，用 `todo!()` 占位
2. **测试**：针对阶段一分析出的公开接口写行为测试
   - 原仓库若有对应 `.test.ts`，直接翻译测试用例（输入/输出/边界
     条件保持一致）
   - 原仓库没测试覆盖的公开行为，根据阶段一的"不变量"自己补测试
3. **实现**：写到测试通过为止
4. **对抗式复核（Adversarial Review）**：实现自测通过后，开一个**全新
   的、不带实现过程上下文的 Claude Code 会话**（或用 `/clear` 清空
   当前会话上下文），只把 diff 喂给它，明确要求它"假设这段代码有
   bug，任务是把 bug 找出来，不是确认代码没问题"。复核者不看实现者
   的推理过程，只看 diff + `PORTING.md` + 原 TS 源码。
   - 复核重点：是否引入了上面"高危陷阱"表里的模式、生命周期/所有权
     是否合理、错误路径是否被正确传播、状态机的事件顺序是否和原版
     一致
   - 复核者提出的问题逐条处理完，且没有新的遗留问题，才算这个模块
     完成，可以进入阶段三验证

#### 代码规范

- 禁止 `.unwrap()` / `.expect()`（测试代码除外）；错误必须通过
  `Result<T, E>` 显式传播，每个 crate 用 `thiserror` 定义自己的错误类型
- 函数参数最多 3 个，超过用 struct 传参（对应 TS 常见的 options
  object 参数）
- trait-first：TS 的 interface（尤其是 Provider 抽象）优先映射为
  Rust trait，不要直接用某个 provider 的具体结构体
- 禁止静默 fallback：TS 里若用 `?? defaultValue` 掩盖了"本不该发生"
  的情况，Rust 侧要显式判断并返回 `Err` 或 panic（视原意图而定），
  不能默默吞掉
- 异步：`async/await` + Promise → `async fn` + `tokio`；流式事件
  （`agent.subscribe`）用 `tokio::sync::mpsc` 或自定义 `Stream`
  trait，不要用回调闭包硬翻译
- **如果需要写一整段注释去解释"这个 workaround 为什么是对的"，说明
  代码本身写错了，应该去修代码，而不是把理由记录下来了事。** 遇到
  编译不过或测试不过就用 `todo!()`/`unimplemented!()`/放宽类型占位
  糊弄过去，属于同一类问题，禁止这样做

#### 明确不逐行复刻的部分

- TUI 渲染层不追求逐行复刻（Ratatui + TEA 架构重写），但**状态机
  语义**（有哪些状态、状态转移条件）必须与原版一致
- Node.js 特定运行时行为（如某些 process/fs API 细节）按 Rust
  生态惯用法重写，不追求 API 名字一致，但要在代码注释或文档里注明
  "此处有意偏离原版，原因是 XXX"

### 阶段三：验证（每个模块写完立刻做，不要攒到最后）

TS 和 Rust 是两个运行时，不能直接跑同一份代码，验证拆成三层：

**1. 单元级：黄金用例对齐**

- 每个翻译过来的测试用例，人工核对 Rust 测试的输入/期望输出与原
  TS 测试文件是否**字面一致**（不是"意思差不多"）
- 数值/字符串类边界条件（空输入、超长输入、Unicode、null vs
  undefined 对应 `None`）逐条列出核对表

**2. 契约级：接口行为对齐**

产出物落盘为每个 crate 目录下的 `CONTRACT_ALIGNMENT.md`（和
`DEVIATIONS.md`、`PORTING_MISTAKES.md` 并列，同一层级），不要只在
对话里写一遍。针对每个公开 API 写对照表：

| 行为场景 | TS 版本行为 | Rust 版本行为 | 是否一致 | 差异原因（如有） |
| -------- | ----------- | ------------- | -------- | ---------------- |

**"是否一致"列若填"否"，必须在"差异原因"列引用 `DEVIATIONS.md` 里
对应的登记条目**（比如"见 DEVIATIONS.md #3"）；不能留一个孤立的
"否"却没有说明——那意味着这是一个还没处理完的行为不一致，不是已经
消化掉的有意偏差。

对有状态机语义的模块（如 agent loop）额外做**事件序列对齐**：给定
同一输入 prompt + mock 的 LLM 响应，原版和 Rust 版产出的事件序列
（event type 顺序）必须相同。

**3. 集成级：跨模块回归**

- 每个 crate 完成后跑：该 crate 全部测试 + `cargo clippy`（零警告）
  - `cargo test --workspace`
- 涉及真实 LLM Provider 网络调用的部分，用录制的 fixture（请求/
  响应 JSON）做离线回放测试，不依赖真实 API Key，保证测试可重复

### 阶段四：收尾对齐（当前阶段——保留已确认的修改）

移植已基本完成。现在的任务不是"重新对照 TS 重写"，而是**核查偏差，
同时不得覆盖或撤销你已手动确认过的修改**。核心矛盾：自动化对齐检查
容易把"你故意改的地方"误判成"没翻译对"，然后试图纠正——这是必须
杜绝的行为。

#### 4.1 偏差日志（`DEVIATIONS.md`，每个 crate 一份，仓库根目录已有则复用）

在写代码或做任何"对齐修正"之前，先检查该 crate 目录下是否已存在
`DEVIATIONS.md`。格式：

| 位置（文件:行/函数名） | 原 TS 行为 | Rust 实际行为 | 修改原因 | 确认状态 |
| ---------------------- | ---------- | ------------- | -------- | -------- |

"确认状态"只有两种取值：`已确认保留` / `待确认`。

#### 4.2 对齐检查流程（强制顺序，禁止跳过第 1 步）

1. **先查日志，再动代码。** 发现 Rust 行为与原 TS 不一致时，先查
   `DEVIATIONS.md` 里是否已有记录：
   - 有记录且状态为"已确认保留" → **禁止修改**，跳过，视为已对齐
     （即使字面上和原 TS 不同）
   - 有记录但状态为"待确认" → 向用户提出，不要自行决定是恢复原版
     还是保留修改
   - 没有记录 → 视为潜在的未识别偏差，走第 2 步
2. **对未记录的偏差分类：**
   - **回归 bug**（明显是翻译遗漏/实现错误，非有意为之）→ 直接修，
     修完照阶段三流程重新跑测试
   - **看起来像有意修改，但没写进日志**（比如代码里能看出改动痕迹，
     但没人补记录）→ 不要擅自判断意图，向用户确认后再决定是否补录
     进 `DEVIATIONS.md`
3. 每次因对齐检查而做出的修改（无论是修 bug 还是补录日志），在
   commit message 或 PR 描述里注明"对齐检查：修正 XXX / 补录偏差 XXX"，
   方便回溯

#### 4.3 移植错误归档（每修一个真 bug 就补一条，不要事后补）

对齐检查中修复的"回归 bug"（不是有意保留的偏差），额外记录到
`PORTING_MISTAKES.md`：

| 位置 | 现象 | 根因模式 | 修复方式 |
| ---- | ---- | -------- | -------- |

"根因模式"尽量归到阶段一"高危陷阱"表里已有的分类（截断/越界/短路
求值/并发顺序等）；如果是新出现的模式，同时把它补回 `PORTING.md`
的陷阱表——这样同一类错误不会在其他文件里重复出现。目的不是留档
邀功，是让下一次遇到同类代码时，复核者能提前认出来。

#### 4.4 与阶段三验证的关系

阶段三的"契约级对齐表""事件序列对齐"现在要改一个判断标准：**不一致
不等于要改**。对照表里"是否一致"这一列，如果差异已经记录在
`DEVIATIONS.md` 且状态为"已确认保留"，填"是（有意偏差，见
DEVIATIONS.md）"，不要标记为"否"然后触发修正。

## 给 Claude Code 的执行提示

- 不确定本次任务范围覆盖哪些 crate 时，先问，不要自行扩大范围
- 三个阶段严格顺序执行：先出分析文档，经确认后再写类型骨架和测试，
  最后才写实现。不要跳过阶段一直接写实现代码
- 遇到 TS 侧的隐式行为（如某处依赖 JS 的类型强转或 `??` 兜底），
  必须在分析文档里显式标注，不要在 Rust 实现里悄悄"合理化"掉
- **进入阶段四（收尾对齐）后：发现任何与原 TS 不一致的地方，第一
  反应是查 `DEVIATIONS.md`，不是直接改代码。** 标记为"已确认保留"
  的偏差是用户明确要保留的东西，禁止在"对齐"名义下悄悄改回去
- 对抗式复核必须真的换一个干净上下文（新会话或 `/clear`），不能
  在同一个已经写完实现、带着"我觉得这段没问题"倾向的会话里自问
  自答式地"复核"一遍
- 修 bug 之前先查一遍"高危陷阱"表和 `PORTING_MISTAKES.md`——很可能
  是同一类模式在别的文件里已经出现过
